## 1. 核心原理修正与概览

在制定策略前，必须厘清 CPython 的真实机制，摒弃常见误区。

| 维度 | List (列表) | Tuple (元组) | 修正说明 |
| :--- | :--- | :--- | :--- |
| **底层结构** | 指向对象的**动态指针数组** | 指向对象的**静态指针数组** | 两者均存储指针。Tuple 长度固定，List 预留扩容空间。 |
| **内存机制** | **Over-allocation**<br>预分配额外空间以均摊 `append` 成本。 | **按需分配**<br>创建时大小即确定，无扩容预留。 | **CPython 无 Tuple Interning**（常量池）。Tuple 的内存优势主要源于无 Over-allocation，而非对象复用。 |
| **语义契约** | **数据集合**<br>关注“同质元素”的“增删改”操作。 | **数据记录**<br>关注“异质字段”的“结构”与“位置”。 | Tuple 类似数据库的一行或 C 的 Struct；List 类似动态数组或缓冲区。 |
| **可变性** | **可变** (Mutable) | **顶层不可变** (Shallow Immutable) | **警惕陷阱**：`t = ([1],)` 是合法的 Tuple，但其内部 List 可被修改。Tuple 仅保证引用地址不变。 |

## 2. 决策矩阵：四维选择法

### 2.1 必要性维度
*   **必须选择 Tuple**：
    *   **字典键 / 集合元素**：`dict` 和 `set` 要求数据结构必须是 Hashable 的。Tuple 仅当其内部所有元素也都 Hashable 时才可哈希。List 不可哈希。
    *   **函数多返回值**：Python 默认将 `return a, b` 打包为 Tuple，这是语言层面的契约。

### 2.2 生命周期维度
*   **选择 List**：数据在生命周期内需要动态变化。
    *   **典型场景**：任务队列、累积结果集、在线用户列表、栈/队列实现。
*   **选择 Tuple**：数据一旦创建便恒定不变（只读）。
    *   **典型场景**：配置信息、常量定义（如 `WEEKDAYS`）、坐标点 `(x, y)`。

### 2.3 数据特征维度
*   **同质集合**：元素类型相同，数量不定，注重遍历 $\rightarrow$ **List**。
*   **异质记录**：元素类型不同，数量固定，注重位置 $\rightarrow$ **Tuple** (推荐升级为 `NamedTuple` 或 `Dataclass`)。

### 2.4 性能维度
*   **大规模数值计算 (>10k)**：**禁用 List/Tuple**，使用 `numpy.ndarray` 或 `array.array` 以获得缓存友好性和内存连续性。
*   **普通规模数据**：**忽略微小的创建速度差异**。优先基于语义（前三个维度）选择，而非为了极其微小的性能提升牺牲代码可读性。

## 3. 详细场景与最佳实践

### 3.1 API 设计规范
在函数接口设计时，明确类型的“能力”而非具体实现。

```python
from typing import Sequence, List, Tuple, Optional

# ✅ 推荐：参数接受抽象协议
def process_items(items: Sequence[int]) -> None:
    """接受 List 或 Tuple，表示内部只读遍历，不修改"""
    for item in items:
        print(item)

# ✅ 推荐：返回值明确语义
def get_config() -> Tuple[str, int]:
    """返回固定结构，建议使用 NamedTuple 增强可读性"""
    return ("localhost", 8080)

def get_active_users() -> List[dict]:
    """返回动态集合，允许调用方增删"""
    return [{"id": 1}, {"id": 2}]
```

### 3.2 数据结构的演进路径
不要过度使用裸 Tuple，随着复杂度增加应及时重构。

1.  **简单记录** $\rightarrow$ `tuple` (如 `(x, y)`)
2.  **字段有语义** $\rightarrow$ `NamedTuple` (如 `Point(x=1, y=2)`)
3.  **需要默认值/方法** $\rightarrow$ `@dataclass(frozen=True)` (保持不可变契约)
4.  **需要验证逻辑** $\rightarrow$ 完整的 `class` (配合 `__slots__` 节省内存)

### 3.3 并发安全与防御性编程
*   **误区**：Tuple 天生线程安全。
*   **真相**：Tuple 仅保证**引用**不可变。若内部包含可变对象（如 List），仍需加锁。
*   **策略**：对于跨线程共享的只读配置，确保 Tuple 内部所有元素均为不可变对象（如 `str`, `int`, `frozenset`）。

## 4. 代码反模式与修正

| 反模式 | 风险 | 修正方案 |
| :--- | :--- | :--- |
| **单元素 Tuple 漏逗号**<br>`t = (1)` | 被解析为整数 `1`，而非 Tuple | **强制加逗号**<br>`t = (1,)` |
| **可变默认参数**<br>`def f(l=[]):` | 默认参数在调用间共享状态 | **使用 None**<br>`def f(l=None): l = l or []` |
| **Tuple 包含可变对象**<br>`t = ([1],)` | `t[0].append(2)` 修改了内容，破坏不可变预期 | **深度不可变**<br>`t = ((1,),)` 或使用 `frozen=True` 的 Dataclass |
| **高频追加 Tuple**<br>`t += (i,)` | 每次 `+=` 创建新对象，O(N) 复杂度，导致 O(N²) 性能灾难 | **使用 List**<br>`l.append(i)` |

---

## 5. 性能评测与陷阱提示

### 5.1 关于评测方法的警告
*   **不要依赖 `sys.getsizeof`**：它仅测量容器本身的大小，不包含内部元素。对于 `[object(), object()]`，List 的额外开销主要体现在其预留的指针空间上。
*   **不要迷信创建速度**：虽然 Tuple 创建通常略快，但在 I/O 密集型或业务逻辑复杂的场景下，这种差异可忽略不计。**不要为了微小的性能提升而牺牲语义的正确性**。

### 5.2 Python 版本敏感性
*   本方案基于 Python 3.8+。
*   **Python 3.11+** 对 List 的内存管理进行了优化，List 与 Tuple 的内存差距有所缩小，但底层的“动态 vs 静态”设计哲学未变。

---

## 6. 总结

**一句话原则**：List 是**动态容器**，用于处理流数据；Tuple 是**静态记录**，用于定义数据契约。

**最终决策清单**：
1.  **需要作为 Key 或 Set 元素？** $\rightarrow$ **Tuple**
2.  **数据结构固定且异构（如记录）？** $\rightarrow$ **Tuple / NamedTuple**
3.  **数据动态变化或同质集合？** $\rightarrow$ **List**
4.  **海量数值计算？** $\rightarrow$ **Array / NumPy**
5.  **不确定时？** $\rightarrow$ 优先考虑 **List**，因为它更灵活；只有在确定需要“不可变性契约”时才降级为 Tuple。
